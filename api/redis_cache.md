# Стратегия кэширования лайв-линии с помощью Redis

Для защиты основной базы данных PostgreSQL от высоких нагрузок (Highload) во время популярных матчей применяется кэширование структуры линии в оперативной памяти (Redis).

## 1. Структура хранения данных в Redis

Данные кэшируются в виде строкового JSON-пакета.

* **Ключ (Key):** `cache:matches:live`
* **Значение (Value):** JSON-массив со списком активных матчей, маркетов и коэффициентов.
* **Время жизни кэша (TTL - Time To Live):** 60 секунд (автоматическое удаление для предотвращения устаревания данных).

## 2. Стратегия инвалидации кэша (Обновление данных)

Чтобы пользователи мгновенно видели актуальные коэффициенты, мы используем гибридную схему обновления:

1. **Write-Through (Синхронное обновление):** Как только букмекерский бэкенд меняет коэффициент или статус матча, система одновременно обновляет запись в PostgreSQL и принудительно перезаписывает ключ `cache:matches:live` в Redis.
2. **Фоновое обновление:** Если по какой-то причине событие от букмекера не пришло, кэш автоматически удалится через 60 секунд (благодаря TTL), и следующий запрос пользователя принудительно считает свежие данные из PostgreSQL и обновит Redis.

## 3. Алгоритм обработки запроса (Схема)

* **Запрос GET `/api/v1/matches`:**
  * Проверить наличие ключа `cache:matches:live` в Redis.
  * **Cache Hit (Кэш найден):** Мгновенно вернуть JSON из Redis (время ответа ~2-5 мс).
  * **Cache Miss (Кэш отсутствует):** Сделать запрос в PostgreSQL -> Сохранить результат в Redis -> Вернуть ответ пользователю.
